# 01 — Data Model

SQLite for the POC. Write everything with SQLAlchemy 2.0 typed models so the move to
PostgreSQL is just a connection string + migration. Use UUID strings for PKs.

## Tables

### `orders`
One row per real merchant order. Multiple emails (confirmation, shipping, refund) map
to the **same** row.

| column              | type        | notes |
|---------------------|-------------|-------|
| `id`                | UUID (PK)   | internal id, generated by us |
| `vendor_name`       | text        | display name ("Zara") |
| `vendor_domain`     | text        | normalized from sender, e.g. `zara.com` — stable |
| `merchant_order_id` | text        | the merchant's order number |
| `purchase_date`     | timestamptz | ISO 8601 |
| `currency`          | text, null  | e.g. `NIS` |
| `total_price`       | numeric, null |  |
| `status`            | enum        | `active`, `shipped`, `partially_returned`, `returned`, `cancelled` |
| `created_at`        | timestamptz | |
| `updated_at`        | timestamptz | |

**Unique constraint:** `(vendor_domain, merchant_order_id)` — this is the dedup anchor.
If `merchant_order_id` is null (some mails omit it), fall back key =
`(vendor_domain, purchase_date::date, total_price)`; if still ambiguous, key on the
source `message_id`. Document whichever fallback fires in `repo.py`.

## Status transitions
- Order confirmation → `active`.
- Shipping mail → `shipped` (no new items, may add tracking later).
- Refund/cancellation mail for an existing order → set affected items to
  `returned`/`cancelled`; recompute order status (`returned` if all, else
  `partially_returned`). Never hard-delete — keep history.

## Definition of done
Models created, `Base.metadata.create_all` produces the schema, a quick pytest inserts
an order with two items and rejects a duplicate `(vendor_domain, merchant_order_id)`.
